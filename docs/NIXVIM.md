# NIXVIM — Mental model

## 1. What Nixvim actually is

Nixvim is a Nix module system for Neovim. You write options; it
generates `init.lua` and a wrapped Neovim package. There is no
"runtime config" — everything is resolved at build time.

The build pipeline:

```
your modules/*.nix
        │
        ▼
nixvim.lib.evalNixvim { system; modules; }   ← new API
        │    (NOT makeNixvimWithModule — that is legacy)
        ▼
config.build.package   ← wrapped neovim in /nix/store
config.build.test      ← smoke-test derivation
```

## 2. The four usage modes

Pick one; they differ only in how the module is imported and which
option root is used.

| Mode               | Option root                  | Use when                       |
|--------------------|------------------------------|--------------------------------|
| NixOS module       | `programs.nixvim.*`          | System-wide nvim on NixOS      |
| Home Manager       | `programs.nixvim.*`          | User nvim via HM               |
| nix-darwin         | `programs.nixvim.*`          | macOS                           |
| Standalone (flake) | (no prefix, options at root) | Per-project, portable flake    |

This repo is standalone. The standalone flake evaluates via
`nixvim.lib.evalNixvim` and exposes the result as a package.
Standalone options are unprefixed (`plugins.x`, `colorschemes.x`),
not under `programs.nixvim`.

## 3. What `plugins.<x>.enable = true` does

Two things, automatically:

1. Adds `<x>` to `extraPlugins` → it lands in the Nixvim-built
   `vim-pack-dir/pack/myNeovimPackages/start/<x>/`. Neovim's
   startup auto-scans `pack/*/start/*` and sources `plugin/<x>.lua`
   for each.

2. Appends `require("<x>").setup(<settings>)` to `extraConfigLua`.
   `settings` is the nix attrset, translated to a lua table.

That is the entire mechanism. `enable = true` is the default
correct path; do not hand-roll a `require` of a Nixvim-managed
plugin unless you specifically need to call `setup` twice or out
of order.

## 4. rtp / packpath flow

The Nixvim wrapper script sets, before `-u init.lua`:

```
--cmd "set packpath^=<vim-pack-dir>"
--cmd "set rtp^=<vim-pack-dir>"
```

Then neovim's startup scans packpath and expands rtp with every
`pack/*/start/<name>/`. So at init time, every `enable = true`
plugin is in the rtp and its `plugin/*.lua` is sourced.

If you introduce a custom plugin manager (lazy.nvim, packer.nvim,
…) on top of this, it will reset rtp/packpath and drop every
Nixvim-managed plugin. See `docs/PLUGINS.md` for the trap.

## 5. `extraConfigLua` layering

Three ordered slots, all strings of raw Lua:

| Slot                | Order | Use for                                     |
|---------------------|-------|---------------------------------------------|
| `extraConfigLuaPre`   | first  | Colorschemes (must be set before anything reads them) |
| `extraConfigLua`      | middle | Plugin `setup()` calls generated by `enable = true` |
| `extraConfigLuaPost`  | last   | Autocmds, final overrides, hand-written requires |

If you need to set a colorscheme, do it in `extraConfigLuaPre` so
plugins that query `vim.g.colors_name` see the right value when
their setup runs.

## 6. Lazy loading

Nixvim has a first-class lazy-loading mechanism: `lz.n`. Set
`plugins.<x>.lazyLoad.enable = true` and configure triggers via
`plugins.<x>.lazyLoad.settings` (cmd, ft, event, keys, etc.).
The generated `setup()` call is routed to `lz.n`'s `after` hook.

**Do not** use lazy.nvim, packer.nvim, or any other external
plugin manager. They reset rtp/packpath and break Nixvim's
plugin loading. See `docs/PLUGINS.md`.

## 7. Pin discipline

Do not add `inputs.nixpkgs.follows = "nixpkgs";` to the nixvim
input. Nixvim is tested against its pinned nixpkgs revision;
using a different nixpkgs causes `"<plugin>" cannot be found in
pkgs` errors. The official flake template does not use `follows`
for this reason.
